# Post-Incident Report (PIR)
**Incident ID:** SIM-IRL-001  
**Severity:** Medium  
**Status:** Closed  
**Date:** (generated at runtime)  

> **Note:** This incident is simulated; telemetry and artifacts are real outputs from AWS CloudTrail/GuardDuty generated by the lab scenario.

## Executive Summary
On 2023-10-27, an IAM user `ir-lab-suspect` performed unauthorized reconnaissance activities and attempted to access restricted S3 data (`canary.txt`). The activity was detected via CloudTrail logs and AccessDenied error codes. The credentials were effectively limited by IAM policy, preventing data exfiltration and persistence.

## Impact
- **Data Confidentiality:** No sensitive data was accessed. The user attempted to read `canary.txt` but was blocked.
- **Service Integrity:** No resources were modified or deleted. `iam:CreateUser` attempts failed.
- **Availability:** No impact.

## Detection
- **Source:** CloudTrail
- **Indicators:**
    - High volume of `Describe*` and `List*` calls (Discovery).
    - `AccessDenied` errors on `s3:GetObject` and `iam:CreateUser`.
    - Activity from a single IAM user `ir-lab-suspect`.

## Root Cause
Simulated compromised credentials for lab purposes. In a real scenario, this would likely stem from leaked access keys or a compromised developer workstation.

## Recovery & Containment Actions
1. **Credentials Rotated:** Access keys for `ir-lab-suspect` were deactivated.
2. **Policy Review:** Verified that the deny policies functioned as expected.

## Recommendations
1. **Enable GuardDuty:** Ensure GuardDuty is active in all regions to catch anomalous discovery tactics automatically.
2. **Alerting:** Create CloudWatch Alarms for `AccessDenied` spikes (>5 in 1 minute).
